## 一、先建立全局观：Agent 到底是什么

一句话：**Agent = LLM（大脑）+ 工具（手脚）+ 记忆（笔记本）+ 循环（反复思考-行动直到完成任务）**。

普通 LLM 调用是「你问一句，它答一句」。Agent 是「你给一个目标，它自己拆解步骤、调用工具、看结果、再决定下一步，循环往复，直到完成」。

最经典的心智模型是 **ReAct 循环**：

```
目标输入
   │
   ▼
┌──────────────────────────────────────┐
│  Thought  （思考：现在该干嘛？）        │
│     │                                  │
│     ▼                                  │
│  Action   （行动：调用某个工具）        │  ← 循环
│     │                                  │
│     ▼                                  │
│  Observation（观察：工具返回了什么）    │
│     │                                  │
│     └──► 够了吗？ ──否──► 回到 Thought  │
└─────────────┬────────────────────────┘
              │是
              ▼
          最终答案
```

记住这张图，后面所有框架（LangChain / AutoGPT / 各种 Agent）本质都是在实现并增强这个循环。

```
阶段0  前置基础         Python + HTTP + JSON + 会调一次 LLM API

  │
阶段2  Agent 核心机制    Function Calling / ReAct / 工具 / 记忆 / 规划

  │
阶段5  多 Agent & 进阶   多智能体协作、反思、评估、防幻觉

**关键心法**：阶段 1、2 一定要**先手写一遍不用框架的版本**，再去用框架。否则框架会变成黑盒，面试一问底层就露馅。

## 三、核心概念词典（必须搞懂的 30 个名词）

| 名词 | 一句话解释 |
|---|---|
| **LLM** | 大语言模型，根据上文预测下一个 token 的概率模型 |
| **Token** | 文本的最小切分单位，模型按 token 计费和计长度 |
| **上下文窗口 (Context Window)** | 模型一次能"看到"的最大 token 数，超了会被截断 |
| **Prompt** | 你喂给模型的输入文本（含 system / user / assistant 角色） |
| **System Prompt** | 设定角色、规则、输出格式的"开场指令" |
| **温度 (Temperature)** | 控制随机性，0=稳定保守，1+=发散有创意 |
| **Few-shot** | 在 prompt 里给几个示例，引导模型模仿格式 |
| **CoT (Chain of Thought)** | 让模型"一步步想"，显著提升推理准确率 |
| **ReAct** | Reasoning+Acting，思考与行动交替的 Agent 范式 |
| **Function Calling / Tool Use** | 模型输出结构化 JSON，由你的代码去执行真实函数 |
| **工具 (Tool)** | Agent 能调用的外部能力（搜索、计算、查库、发邮件…） |
| **Agent Loop** | 思考→行动→观察的循环，直到任务完成或达到上限 |
| **记忆 (Memory)** | 短期=当前对话历史；长期=存到外部库可跨会话检索 |
| **规划 (Planning)** | 把大目标拆成有序子任务（如 Plan-and-Execute） |
| **反思 (Reflection)** | Agent 检查自己的输出并改进（Reflexion / Self-critique） |
| **RAG** | 检索增强生成：先查资料再让模型基于资料回答 |
| **Embedding** | 把文本变成一串数字向量，语义相近的向量距离也近 |
| **向量数据库** | 存向量并支持"最近邻"检索（FAISS/Chroma/Milvus/pgvector） |
| **分块 (Chunking)** | 把长文档切成小段再做 embedding |
| **召回 (Retrieval)** | 根据问题向量找出最相关的若干文档块 |
| **重排 (Re-rank)** | 对召回结果用更精细的模型再排一次序 |
| **幻觉 (Hallucination)** | 模型一本正经地编造不存在的事实 |
| **结构化输出** | 强制模型输出固定 JSON Schema，方便程序解析 |
| **MCP** | Model Context Protocol，给模型接工具/数据的开放协议 |
| **多智能体 (Multi-Agent)** | 多个角色 Agent 分工协作（如产品/程序员/测试） |
| **Token 成本** | 输入+输出 token × 单价，Agent 多轮循环很费 token |
| **Guardrails** | 护栏：输入/输出校验，防越权、防注入、防有害内容 |
| **Prompt 注入** | 攻击者通过输入操纵模型，绕过你的指令 |
| **可观测性** | 记录每一步的 prompt/响应/工具调用，便于排错（如 LangSmith） |
| **评估 (Eval)** | 用数据集和指标量化 Agent 好不好（准确率/通过率等） |

> 学习建议：把这张表打印出来，每学一个阶段就回来勾掉你能"用自己的话讲给别人听"的词。

## 四、分阶段详解（学什么 + 怎么验证 + 项目）

### 阶段 0：前置基础（1-3 天）
- **学什么**：Python 基础（函数、类、字典、`requests`/`httpx`）、JSON、环境变量、虚拟环境（venv）、会读 API 文档。
- **怎么验证**：能用 Python 发一个 HTTP 请求并解析返回的 JSON。
- **项目**：注册一个大模型 API（如 OpenAI / Anthropic / DeepSeek / 通义千问），写 10 行代码成功调用一次，打印回复。

### 阶段 1：LLM 与 Prompt（3-5 天）
- **学什么**：
  - system / user / assistant 三种角色
  - temperature、max_tokens、流式输出（stream）
  - 提示词技巧：角色设定、Few-shot、CoT、让它输出 JSON
  - 上下文窗口的概念和"对话历史"如何拼接
- **怎么验证**：能让模型稳定输出**合法 JSON**（用 `json.loads` 解析不报错）。
- **项目**：写一个命令行聊天机器人，支持多轮对话（把历史消息存进列表传回去）。

### 阶段 2：Agent 核心机制（1-2 周，最重要）
- **学什么**：
  1. **Function Calling**：定义工具的 JSON Schema → 模型决定调哪个 → 你的代码执行 → 把结果喂回去。
  2. **ReAct 循环**：手写一个 while 循环实现"思考-行动-观察"。
  3. **工具设计**：计算器、网页搜索、读写文件、查数据库。
  4. **记忆**：短期（对话列表）+ 长期（存文件/库）。
  5. **规划**：复杂任务先让模型列 to-do 再逐条执行。
- **怎么验证**：**不依赖任何 Agent 框架**，纯手写一个能调用 2-3 个工具、能多步完成任务的 Agent。
- **项目**：见下面【hello-agent】。

### 阶段 3：RAG 检索增强（1 周）
- **学什么**：文档加载 → 分块 → Embedding → 存向量库 → 相似检索 → 拼进 prompt → 生成答案。再进阶：重排、混合检索（关键词+向量）。
- **怎么验证**：丢给它一份它"没见过"的 PDF，它能基于 PDF 内容准确回答，并能说"资料里没有"。
- **项目**：做一个"和你的 PDF 对话"的问答机器人。

### 阶段 4：框架实战（1 周）
- **学什么**：用 **LangChain** 或 **LlamaIndex** 把前面手写的东西重写一遍，体会框架帮你省了什么。重点看：Tools、Agent、Memory、Retriever、Chain/Graph 这几个抽象。
- **怎么验证**：能说清"框架的 Agent 内部其实就是我手写的那个 ReAct 循环"。
- **项目**：用 LangGraph 把 hello-agent 重构成图（节点+边）形式。

### 阶段 5：多 Agent & 进阶（1-2 周）
- **学什么**：多智能体协作模式（如 AutoGen / CrewAI / MetaGPT 的思路）、反思机制、Agent 评估、防幻觉、防 prompt 注入。
- **项目**：做一个"产品经理 + 程序员 + 测试"三角色协作写小程序的多 Agent demo。

### 阶段 6：工程化（持续）
- **学什么**：token 成本与延迟优化（缓存、批处理、小模型分流）、可观测（日志/trace）、错误重试与降级、安全护栏、部署（Docker）。
- **项目**：给你的 Agent 加上：成本统计、失败重试、主模型挂了自动切备用模型。

## 五、动手项目阶梯（hello-agent → 进阶）

按顺序做，每个都比上一个多一点东西。**前 3 个不要用框架，纯手写。**

1. **hello-llm**（阶段1）：命令行多轮聊天机器人。
2. **hello-tool**（阶段2）：给模型加一个"计算器"工具，问它"123*456 等于几"，它会调用工具而不是瞎算。
   - 核心：定义工具 schema → 模型返回要调的工具和参数 → 你执行 → 结果回传 → 模型给最终答案。
3. **hello-agent**（阶段2，里程碑）：一个能"思考-行动-观察"循环、带 2-3 个工具（计算器 + 搜索 + 当前时间）的 ReAct Agent。能完成"现在北京几点？再帮我算下到春节还有几天"这种需要多步+多工具的任务。
4. **chat-with-pdf**（阶段3）：RAG 问答，和你的文档对话。
5. **research-agent**（阶段4）：用框架做一个"自动搜资料 → 总结 → 写报告"的研究助手。
6. **multi-agent-team**（阶段5）：多角色协作完成一个小任务。
7. **production-agent**（阶段6）：加上记忆持久化、成本统计、重试降级、Docker 部署。

> 你现在电脑里的 `AIread` 项目其实就是一个很好的"阶段6"参考：它有采集→清洗→LLM 打分→精选→讲解的完整管线，还有主模型失败自动降级、token 统计、Docker 部署。学到阶段 4-6 时可以回去读它的源码当案例。

### hello-agent 最小骨架（伪代码，帮你建立直觉）

```python
tools = {
    "calculator": lambda expr: str(eval(expr)),
    "now":        lambda _:    datetime.now().isoformat(),
}

messages = [{"role": "system", "content": "你是一个会使用工具的助手。需要时调用工具。"}]
messages.append({"role": "user", "content": 用户问题})

for step in range(MAX_STEPS):              # ← Agent Loop，防死循环要有上限
    resp = llm(messages, tools=工具的schema)   # 模型思考
    if resp 想调用工具:                      # Action
        result = tools[工具名](参数)          # 真正执行
        messages.append(工具调用记录)
        messages.append({"role": "tool", "content": result})  # Observation 回传
        continue                            # 回到循环顶部继续想
    else:
        return resp.内容                     # 模型觉得够了，给最终答案
```

把这段读懂、自己实现出来，你就跨过了 Agent 最重要的门槛。

## 六、八股面试题（分类 + 答案要点）

> 用法：先盖住答案自己说，再对照要点。能讲出**为什么**比背概念更重要。

### A. LLM 基础

**Q1. 什么是 token？为什么重要？**
要点：文本的最小切分单位（不完全等于字/词）；模型按 token 计长度和计费；上下文窗口、成本、延迟都和 token 数挂钩。

**Q2. temperature 和 top_p 是什么？**
要点：都控制输出随机性。temperature 缩放概率分布（越高越随机）；top_p（核采样）只在累计概率前 p 的候选里采样。要稳定/可解析输出就调低；要创意就调高。一般二选一调。

**Q3. 上下文窗口满了怎么办？**
要点：截断旧消息、做摘要压缩历史、用 RAG 只取相关片段、滑动窗口、把长期信息外置到记忆库。

**Q4. 为什么会有幻觉？怎么缓解？**
要点：模型本质是概率预测、训练数据有限/过时、没有事实校验。缓解：RAG 提供事实依据、要求引用来源、降低 temperature、让模型"不知道就说不知道"、加事实校验/二次核对。

### B. Prompt 工程

**Q5. Zero-shot / Few-shot / CoT 区别？**
要点：zero-shot 不给例子直接问；few-shot 给几个示范；CoT 让模型显式"一步步推理"再给答案，对数学/逻辑题提升明显。

**Q6. 怎么让模型稳定输出 JSON？**
要点：system 里明确 schema 和"只输出 JSON"、给 few-shot 例子、用 Function Calling / 结构化输出 / JSON mode、代码侧做容错解析（抠出 `{}` 部分、校验、失败重试）。

**Q7. 什么是 prompt 注入？怎么防？**
要点：用户输入里藏指令操纵模型（如"忽略上面所有指令"）。防：区分可信指令与不可信数据、对用户输入做围栏、输出做校验、最小权限工具、敏感操作要人确认。

### C. Agent 核心机制（重灾区，必背）

**Q8. Agent 和普通 LLM 调用的区别？**
要点：普通调用是单轮问答；Agent = LLM + 工具 + 记忆 + 循环，能自主拆解任务、调用工具、根据结果多步决策直到完成目标。

**Q9. 讲讲 ReAct。**
要点：Reasoning + Acting 交替。Thought（思考）→ Action（调工具）→ Observation（看结果）循环，直到得出答案。优点：把推理和工具结合，过程可解释、可纠错。

**Q10. Function Calling 的完整流程？**
要点：① 你定义工具的 JSON Schema 一起传给模型 → ② 模型判断要不要调、调哪个、参数是什么（返回结构化调用，不是它自己执行）→ ③ 你的代码执行真实函数 → ④ 把结果作为新消息回传 → ⑤ 模型基于结果继续或给最终答案。**关键：模型只"决定"，执行永远在你的代码里。**

**Q11. Agent 的记忆怎么设计？**
要点：短期记忆=当前对话历史（放进上下文）；长期记忆=存外部（向量库/数据库），需要时检索召回。还有按重要性/时效做摘要、遗忘策略。

**Q12. 怎么防止 Agent 陷入死循环 / 失控？**
要点：设最大步数/最大 token/超时；检测重复动作；每步成本预算；关键操作加人工确认（human-in-the-loop）；失败兜底返回。

**Q13. Planning 类 Agent（Plan-and-Execute）和 ReAct 区别？**
要点：ReAct 边想边做、走一步看一步；Plan-and-Execute 先一次性规划出完整步骤再逐步执行，适合复杂可拆解任务，过程更可控但不如 ReAct 灵活应变。常结合使用（规划+执行中再 ReAct）。

**Q14. 什么是 Reflection / 反思机制？**
要点：让 Agent 对自己的输出做自我批评和修正（如 Reflexion）。生成→评估→根据反馈重做，提升正确率，代价是更多 token 和延迟。

### D. RAG

**Q15. 完整讲一遍 RAG 流程。**
要点：离线—文档加载→分块→Embedding→存向量库；在线—问题 Embedding→向量检索 Top-K→（可选重排）→把片段拼进 prompt→LLM 基于片段生成答案（并可标注来源）。

**Q16. 为什么要分块？块大小怎么定？**
要点：文档太长放不进上下文、且整篇 embedding 语义太杂检索不准。块太大噪声多、块太小丢上下文，常用几百 token + 重叠（overlap）保证语义连续，按文档类型调。

**Q17. Embedding 是什么？相似度怎么算？**
要点：把文本映射成定长向量，语义相近向量也近。常用余弦相似度 / 点积 / 欧氏距离衡量。

**Q18. RAG 检索不准怎么优化？**
要点：更好的分块、混合检索（关键词 BM25 + 向量）、加重排模型、query 改写/扩展、多路召回、元数据过滤、调 Top-K、换更强 embedding 模型。

**Q19. RAG vs 微调（Fine-tuning）怎么选？**
要点：RAG 适合知识频繁更新、要可溯源、成本低、即插即用；微调适合改变模型"风格/能力"、固定领域、对延迟敏感。常 RAG 优先，必要时两者结合。

### E. 多 Agent & 工程

**Q20. 为什么要多 Agent？有什么代价？**
要点：分工专精、可并行、模块清晰（如规划/执行/审查分离）。代价：协调复杂、token 成本和延迟翻倍、易出现互相甩锅/发散，需要好的编排和终止条件。

**Q21. 怎么评估一个 Agent 好不好？**
要点：任务成功率/通过率、答案准确率、工具调用正确率、步数/成本/延迟、人工评分、用专门 eval 数据集和 LLM-as-judge。要建可量化、可回归的评测集。

**Q22. 怎么控制 Agent 的成本和延迟？**
要点：用小模型做简单步骤、缓存（prompt/结果）、减少不必要的循环和上下文、批处理、流式输出改善体感、限制工具调用次数、主贵备便宜的模型路由。

**Q23. 生产环境 Agent 还要考虑什么？**
要点：可观测（trace 每步）、错误重试与降级、超时、限流、安全护栏与权限、隐私脱敏、prompt 版本管理、A/B 与回归评测、监控成本。

**Q24. MCP 是什么？**
要点：Model Context Protocol，一个开放协议，标准化"模型如何连接外部工具和数据源"，让工具能被不同客户端/模型复用，类似 Agent 世界的 USB 接口。

### F. 高频"项目深挖"问题（面试官会顺着你简历问）
- 你这个 Agent 用了哪些工具？工具调用失败怎么处理？
- 上下文超长你怎么解决的？
- 怎么保证输出格式稳定？解析失败怎么办？
- 成本大概多少？怎么降的？
- 遇到幻觉/答错怎么定位和改进？
- 为什么用 ReAct 而不是直接 Plan-and-Execute？

> 准备方法：把你做的 hello-agent / chat-with-pdf 项目，针对上面每个问题都准备一段"我当时怎么做的"，这比背八股有用 10 倍。
