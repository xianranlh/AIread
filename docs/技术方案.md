# AI 技术雷达平台 · 技术方案

> 自动拉取 GitHub 热门项目与 AI 最新技术，由 AI 生成讲解与扩展学习内容的平台
> 定位：个人学习 / MVP 原型 · AI 策略：混合方案 · 日期：2026-06

---

## 1. 项目概述

### 1.1 核心功能

| 功能 | 说明 |
|---|---|
| 自动采集 | 每日定时拉取 GitHub Trending、AI 论文、技术资讯 |
| 智能筛选 | AI 去重、打分、分类，过滤噪音，留下真正值得看的 |
| 深度讲解 | 对精选项目生成结构化讲解：是什么、解决什么问题、技术原理、上手路径 |
| 扩展学习 | 关联相似项目、前置知识、延伸阅读，形成知识网络 |
| 多端呈现 | Web 站点 + RSS 订阅，可选邮件/Telegram 推送 |

### 1.2 MVP 边界

**做**：每日自动管线、中文讲解、静态站点展示、历史归档可搜索。
**不做**（留到后期）：用户系统、评论、实时更新、交互式问答（Phase 3 再加）。

### 1.3 设计原则

1. **零/低成本优先**：调度用 GitHub Actions（免费），托管用 Vercel/Cloudflare Pages（免费），AI 用「便宜模型粗筛 + 强模型精讲 + Batch API 半价」。
2. **管线与展示解耦**：采集-处理-生成是离线批处理，前端只读静态数据，互不阻塞，故障隔离。
3. **先跑通再优化**：SQLite 起步，不上消息队列、不上微服务。

## 2. 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                  调度层（GitHub Actions cron，每日 2 次）         │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ ① 采集层 Collectors（Python）                                  │
│   GitHub Trending 爬虫 │ GitHub Search API │ HN Algolia API   │
│   arXiv API │ HF Trending Papers/Models │ 官方博客 RSS          │
└──────────────────────────────┬──────────────────────────────┘
                               ▼  原始条目（JSON）
┌─────────────────────────────────────────────────────────────┐
│ ② 处理层 Pipeline                                             │
│   规则清洗 → 指纹去重 → 向量去重(embedding) → AI 打分分类(Haiku)   │
└──────────────────────────────┬──────────────────────────────┘
                               ▼  精选 Top-N 条目
┌─────────────────────────────────────────────────────────────┐
│ ③ AI 讲解引擎                                                  │
│   抓取 README/论文摘要 → 结构化 Prompt → Sonnet(Batch) 生成讲解   │
│   → 关联扩展（向量检索相似条目 + 前置知识图）                       │
└──────────────────────────────┬──────────────────────────────┘
                               ▼  写入
┌─────────────────────────────────────────────────────────────┐
│ ④ 存储层  SQLite（条目/讲解/向量 sqlite-vec）→ 导出 JSON/MD       │
└──────────────────────────────┬──────────────────────────────┘
                               ▼  构建时读取
┌─────────────────────────────────────────────────────────────┐
│ ⑤ 展示层  Astro 静态站点（Vercel/CF Pages）+ RSS + 站内搜索       │
│   可选推送：邮件(Resend) / Telegram Bot                         │
└─────────────────────────────────────────────────────────────┘
```

**一次完整管线**：Actions 触发 → 采集 ~100-200 条原始数据 → 去重筛选出 Top 15-30 → AI 生成讲解 → 提交回仓库（数据即代码）→ 触发前端重新构建部署。全程 10-20 分钟，无需任何常驻服务器。

## 3. 技术选型总览

| 层 | 选型 | 理由 |
|---|---|---|
| 语言 | Python 3.12 | 爬虫/数据/AI 生态最全，个人开发效率最高 |
| 采集 | httpx + BeautifulSoup + feedparser | 轻量够用，无需 Scrapy |
| 调度 | GitHub Actions cron | 免费、自带日志和重试、数据可直接 commit 回仓库 |
| 数据库 | SQLite + sqlite-vec | 单文件零运维，向量检索够用；后期平滑迁 PostgreSQL+pgvector |
| AI-粗筛 | Claude Haiku 4.5（Batch） | $1/$5 每百万 token，批量半价，打分分类绰绰有余 |
| AI-精讲 | Claude Sonnet 4.6（Batch） | $3/$15，讲解质量与成本的最佳平衡点 |
| AI-兜底/实验 | DeepSeek V4 Flash 或 Ollama + Qwen3 本地 | 开源便宜，做对照实验、降级备份 |
| Embedding | BGE-M3（本地跑，免费） | 中英混合检索效果好，CPU 可推理 |
| 前端 | Astro + Tailwind CSS | 内容型站点首选：默认零 JS、构建快、Markdown 原生支持 |
| 托管 | Vercel 或 Cloudflare Pages | 免费额度对个人项目绰绰有余 |
| 推送(可选) | Resend(邮件) / Telegram Bot API | 免费层够用 |

**为什么不是 Next.js + 常驻后端？** MVP 阶段没有动态交互需求，静态生成 = 零服务器成本 + 零运维 + 天然抗崩。Phase 3 需要交互问答时再加一个 FastAPI 服务即可，架构不用推倒。

## 4. 数据采集层

### 4.1 GitHub 热门项目（三路并采）

| 方案 | 说明 | 限制 |
|---|---|---|
| **A. 爬取 github.com/trending** | 官方无 Trending API，直接解析 HTML（结构多年稳定），按 daily/weekly + 语言维度抓 | 频率别太高，每日 2 次完全安全 |
| **B. GitHub Search API（官方）** | `created:>{7天前} sort:stars` 抓新晋高星项目，补充 Trending 看不到的「快速崛起」项目 | 认证后 30 req/min，足够 |
| **C. OSS Insight Public API** | TiDB 团队基于 GHArchive 的免费 API，有现成 `/trends/repos` 端点 | beta，600 req/h/IP |

A 为主，B 补盲区，C 做容灾备份。每个 repo 额外用官方 API 拉取：stars 增速、README、语言、topics、最近 release。

### 4.2 AI 技术资讯源

| 来源 | 接口 | 抓什么 |
|---|---|---|
| Hacker News | Algolia HN Search API（免费无需 key） | 搜 AI/LLM 关键词 + 前页高分帖 |
| arXiv | 官方 API / RSS（cs.AI, cs.CL, cs.LG） | 当日新论文，标题+摘要 |
| Hugging Face | `huggingface_hub` 库 + Trending Papers 页 | 趋势模型/数据集/论文（注：Papers with Code 已于 2025-07 关停，HF Trending Papers 是其官方继任者） |
| 官方博客 RSS | OpenAI / Anthropic / DeepMind / Meta AI 等 | 重大发布第一手信息 |
| Reddit（可选） | r/LocalLLaMA, r/MachineLearning JSON 接口 | 社区风向 |

每个来源写成一个独立 Collector 类（统一接口 `fetch() -> list[RawItem]`），单源挂掉不影响整体，新增数据源只加一个文件。

```python
class RawItem(BaseModel):
    source: str          # "github_trending" | "hn" | "arxiv" | ...
    title: str
    url: str
    summary: str | None
    metrics: dict        # stars/points/upvotes 等原始热度
    fetched_at: datetime
```

## 5. 数据处理层（省钱的关键）

> 原则：能用规则解决的不用 AI，能用便宜模型的不用贵模型。

1. **规则清洗**：黑名单域名、awesome-list 类仓库降权、非 AI 相关语言过滤。
2. **指纹去重**：URL 规范化 + 标题 simhash，跨源去重（同一项目常同时出现在 Trending 和 HN）。
3. **向量去重**：BGE-M3 算 embedding，与近 14 天历史条目余弦相似度 > 0.92 视为旧闻，丢弃。**解决「天天都是同几个项目」的核心痛点。**
4. **AI 打分分类（Haiku 4.5 Batch）**：一次请求批量评 20 条，输出 JSON：

```json
{"id": 17, "score": 8.5, "category": "推理框架",
 "tags": ["LLM", "量化", "边缘部署"],
 "reason": "解决大模型在消费级显卡上的部署问题，star 增速异常快",
 "worth_deep_dive": true}
```

5. **排序截断**：综合分 = AI 分 × 热度增速权重 × 新颖度，取 Top 15-30 进入讲解环节。

## 6. AI 讲解引擎（核心价值所在）

### 6.1 混合模型分工

| 任务 | 模型 | 调用方式 |
|---|---|---|
| 打分/分类/标签/翻译标题 | Haiku 4.5 | Batch API（半价） |
| 深度讲解、周报综述 | Sonnet 4.6 | Batch API（半价，每日管线不需要实时） |
| 向量化 | BGE-M3 本地 | 免费 |
| 降级备份/对照实验 | DeepSeek V4 Flash（API）或 Ollama+Qwen3（本地） | 主 API 故障时自动切换 |

### 6.2 讲解生成流程

1. **取证**：抓取 README 全文（截断至 ~15K token）、repo 元数据、相关 HN 讨论高赞评论、论文摘要。**讲解必须基于真实材料生成，杜绝模型凭空发挥。**
2. **结构化输出**：要求模型按固定 schema 输出 JSON，直接入库渲染：

```json
{
  "one_liner": "一句话：它是什么",
  "problem": "解决什么问题，之前的方案差在哪",
  "how_it_works": "核心技术原理（面向工程师的白话）",
  "highlights": ["亮点1", "亮点2"],
  "getting_started": "10 分钟上手路径",
  "prerequisites": ["需要先懂的概念，附一句话解释"],
  "related": ["值得对比的同类项目"],
  "verdict": "适合谁用 / 现在该不该投入学习",
  "confidence_notes": "材料中没说清、可能不准的点"
}
```

3. **扩展学习**：用向量检索从历史库中找出相似条目和前置知识条目，生成「学习路径」链接，让平台越用越像知识网络而非新闻流。
4. **每周综述**：周日由 Sonnet 把本周所有条目汇总成一篇「本周 AI 趋势」长文（这是单条讲解之外的第二个核心价值点）。

### 6.3 防幻觉措施

- Prompt 中强制「只基于提供的材料，材料没有的写 `confidence_notes`」；
- 讲解页展示原始链接 + 「AI 生成，请核实」标识；
- 对输出 JSON 做 schema 校验（Pydantic），字段缺失自动重试一次。

## 7. 存储设计

SQLite 单文件（随仓库 commit 即自动备份），四张表：

```sql
items(id, source, title, url, summary, metrics_json, category,
      tags_json, score, status, created_at)           -- 原始+筛选结果
explanations(item_id, content_json, model, tokens_used, created_at)
embeddings(item_id, vector BLOB)                       -- sqlite-vec 虚拟表
weekly_digests(week, content_md, created_at)
```

管线结束时导出 `data/YYYY-MM-DD.json` 供 Astro 构建消费。数据量预估：每天 30 条精选 ≈ 1 万条/年，SQLite 轻松应对；若 Phase 3 需要多用户并发，`pgloader` 一条命令迁 PostgreSQL。

## 8. 部署与运行

> **更新（2026-06-03）**：实际实现已改为 **自有 VPS + Docker Compose** 部署（web + scheduler 两个容器，APScheduler 替代 GitHub Actions 做调度，FastAPI 服务端渲染替代 Astro 静态站）。代码与部署步骤见 `ai-tech-radar/README.md`。以下 GitHub Actions 方案保留作为零成本备选。

### 8.1 仓库结构（monorepo）

```
ai-tech-radar/
├── pipeline/            # Python 管线
│   ├── collectors/      # 每个数据源一个文件
│   ├── processors/      # 清洗、去重、打分
│   ├── explainer/       # 讲解生成 + prompts/
│   └── main.py
├── web/                 # Astro 前端
├── data/                # SQLite + 导出 JSON（commit 进仓库）
└── .github/workflows/daily.yml
```

### 8.2 GitHub Actions 工作流

```yaml
on:
  schedule: [{cron: "0 1,13 * * *"}]   # UTC，约北京时间 9 点 / 21 点
  workflow_dispatch:                    # 手动触发用于调试
jobs:
  pipeline:
    runs-on: ubuntu-latest
    steps:
      - checkout → 装依赖 → python pipeline/main.py
      - commit data/ 回仓库            # 触发 Vercel 自动重新部署
```

密钥（ANTHROPIC_API_KEY、GITHUB_TOKEN）放 Actions Secrets。免费额度：公开仓库 Actions 不限时长，私有仓库 2000 分钟/月，每日两次 × 15 分钟 ≈ 900 分钟/月，够用。

## 9. 成本估算（每月）

| 项 | 用量假设 | 月成本 |
|---|---|---|
| Haiku 4.5 粗筛 | 150 条/天 × ~0.5K 入 + 0.1K 出，Batch 半价（$0.5/$2.5 每 M） | ≈ $2 |
| Sonnet 4.6 精讲 | 15 条/天 × (8K 入 + 2K 出) + 每周综述，Batch 半价（$1.5/$7.5 每 M） | ≈ $12 |
| Embedding | BGE-M3 本地（Actions runner CPU 上跑） | $0 |
| 调度/托管/CDN | GitHub Actions + Vercel 免费层 | $0 |
| 域名（可选） | — | ≈ $1 |
| **合计** | | **≈ $15/月** |

弹性开关：精讲加到 30 条/天 ≈ $25/月；降到 10 条/天 ≈ $10/月；精讲换 DeepSeek V4 Flash 可压到 $3 以内（质量略降，建议先用 Sonnet 建立质量基线再做对照）。

## 10. 实施路线图

| 阶段 | 周期 | 交付物 |
|---|---|---|
| **P0 管线验证** | 第 1 周 | 跑通「GitHub Trending + HN 采集 → Haiku 筛选 → Sonnet 讲解 → 输出 Markdown」，先不做前端，产物直接看 MD 文件 |
| **P1 上线 MVP** | 第 2-3 周 | Astro 站点 + Actions 自动化 + RSS；接入 arXiv、HF 数据源；向量去重 |
| **P2 体验完善** | 第 4-6 周 | 站内搜索、分类/标签页、每周综述、学习路径关联、邮件/TG 推送 |
| **P3 交互升级（可选）** | 之后 | FastAPI 后端 + 流式问答：「针对这个项目继续提问」（RAG 注入 README），引入用户收藏 |

**P0 的验收标准**：连续 3 天自动产出你自己愿意每天看的日报——内容质量是这个项目唯一的生死线，先验证它。

## 11. 风险与注意事项

| 风险 | 应对 |
|---|---|
| GitHub Trending 页面改版 | 三路并采（爬虫/Search API/OSS Insight），监控解析失败告警 |
| API 限流 | 所有请求带退避重试；GitHub 用认证 token（5000 req/h） |
| AI 幻觉损害可信度 | 基于材料生成 + confidence_notes + 原文链接（见 6.3） |
| 版权 | 摘要+讲解+注明出处属合理使用范畴；不全文转载论文/博客 |
| 内容同质化（与现有日报雷同） | 差异点押在「结构化深讲 + 前置知识 + 学习路径」，而非罗列链接 |
| 模型 API 涨价/故障 | LLM 调用封装统一接口，DeepSeek/Ollama 一键切换 |
